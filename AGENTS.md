# AGENTS.md

## 1. 目的

本文件用于规范 AI agents（如 Codex）与协作者在本仓库中的工作方式。

本项目是一个量化选基系统。我们希望所有改动都符合以下目标：

- 可复现
- 可测试
- 可审计
- 可持续迭代
- 避免量化研究中的常见偏差
- 保持研究逻辑与工程实现的一致性

Agent 在本仓库中的首要目标不是“尽快写出能跑的代码”，而是：

1. 保证金融逻辑正确
2. 保证回测口径严谨
3. 保证改动可验证、可追踪
4. 保证代码结构清晰、便于长期维护

## 2. 项目概述

### 2.1 项目目标

本项目用于基金筛选、评分、组合构建、回测与评估，典型流程包括：

1. 获取并更新基金相关数据
2. 构建基金池（universe）
3. 计算基金因子 / 特征
4. 对基金进行打分与排序
5. 构建基金组合
6. 执行历史回测
7. 输出绩效评估与研究报告

### 2.2 当前工作重点

以下内容通常是优先级最高的工程方向：
- 标准化数据层
- 可复现的因子计算
- 统一的回测框架
- 可配置的策略参数管理
- 实验结果追踪与对比
- 回归测试与结果稳定性验证

### 2.3 非目标

除非明确要求，以下内容不属于本项目默认范围：

- 实盘下单与交易执行
- 高频 / 分钟级策略
- 非基金资产的完整交易系统
- 依赖人工修改数据文件的流程
- 无法复现的“经验性结果修补”

## 3. 仓库结构约定

以下为推荐目录职责。若当前仓库结构与此不同，agent 应优先遵循现有结构，并在必要时提出结构优化建议，而不是擅自大幅重组。

```text
.
├─ src/
│  ├─ data_ingestion/     # 数据获取、更新、落库
│  ├─ data_processing/    # 数据清洗、标准化、对齐
│  ├─ universe/           # 基金池构建逻辑
│  ├─ features/           # 因子/特征计算
│  ├─ ranking/            # 打分、排序、筛选
│  ├─ portfolio/          # 组合构建与权重分配
│  ├─ backtest/           # 回测引擎
│  ├─ evaluation/         # 绩效指标与分析
│  ├─ reporting/          # 报表与可视化输出
│  └─ common/             # 通用工具、配置、日志等
├─ configs/               # 配置文件
├─ tests/                 # 单元/集成/回归测试
├─ research/              # 研究型 notebook，仅用于探索
├─ docs/                  # 设计文档、方法说明
├─ scripts/               # CLI 脚本、批处理入口
├─ data/                  # 数据目录（若保留在仓库外，应在 README 中说明）
├─ outputs/               # 回测结果、实验输出、报表
└─ AGENTS.md
```

### 3.1 目录职责

- `src/`：生产代码唯一可信来源
- `research/`：探索性分析，不作为生产逻辑来源
- `configs/`：策略参数、数据源、实验参数等
- `tests/`：所有可验证行为的测试
- `outputs/`：运行结果，不应反向依赖到 `src/`

### 3.2 明确约束

- notebook 中验证通过的逻辑，若需要长期使用，必须迁移到 `src/`
- 生产逻辑不应隐藏在 notebook 单元中
- 不应将临时脚本散落在仓库根目录
- 不应把输出结果文件写入源码目录

## 4. 领域定义

Agent 在修改策略、数据或回测相关逻辑前，必须理解以下术语。

### 4.1 Universe（基金池）

某一调仓时点允许参与筛选的基金集合。

### 4.2 Rebalance Date（调仓日）

执行基金筛选、评分、组合调整的日期。

### 4.3 Signal Date（信号日）

生成因子与排名信号的日期。必须明确与调仓执行日期之间的关系。

### 4.4 Execution Date（执行日）

组合权重真正生效的日期。回测中必须明确。

### 4.5 Factor（因子）

用于刻画基金收益、风险、风格、稳定性、拥挤度或管理能力的可计算指标。

### 4.6 Benchmark（基准）

用于比较策略表现的基准组合或指数。

### 4.7 Look-ahead Bias（前视偏差）

在某时点使用了当时不可获得的未来信息。严格禁止。

### 4.8 Survivorship Bias（生存者偏差）

仅保留当前仍存在的基金样本，导致历史回测失真。应尽量避免。

### 4.9 Data Lag（数据滞后）

某些数据虽属于过去时点，但在实际披露和可获得时间上存在延迟。必须纳入策略设计。

## 5. 工程原则

以下原则优先于“代码简短”“实现速度快”。

### 5.1 正确性优先于性能

若正确性、可解释性、可复现性与性能冲突，优先保证前三者。

### 5.2 可复现优先

同样的代码版本、数据版本、配置文件，应尽可能得到一致结果。

### 5.3 严禁前视偏差

任何特征、标签、选基结果、回测收益，必须严格遵守时间边界。

### 5.4 尽量避免生存者偏差

基金池构建应尽可能考虑历史上已清盘、合并或不再存续的基金。

### 5.5 配置驱动

窗口长度、调仓周期、过滤阈值、评分权重、风险约束等参数应放在配置中，而不是散落在代码里硬编码。

### 5.6 分层清晰

数据、特征、评分、组合、回测、评估应尽量分层，避免单个函数承担过多职责。

### 5.7 小步迭代

Agent 应优先提交最小可验证改动，而不是一次性做大规模重构。

### 5.8 向后兼容

如果改动会影响历史实验的可比性，必须明确说明，并尽量保留旧逻辑入口或迁移说明。

## 6. Agent 行为边界

### 6.1 Agent 可以做的事

在无需额外确认的情况下，Agent 可以：

- 重构代码以提升可读性和模块化
- 将稳定的 notebook 逻辑迁移到 `src/`
- 增加缺失的测试
- 补充文档
- 将硬编码参数配置化
- 修复明确的、可验证的 bug
- 改善日志、异常处理、类型标注

### 6.2 Agent 必须先确认再做的事

以下操作必须先征求用户确认：

- 修改默认策略逻辑
- 修改基金池定义
- 修改因子定义或打分口径
- 修改回测信号时点 / 执行时点
- 修改绩效指标计算方式
- 修改默认 benchmark
- 引入新的第三方数据源
- 删除历史实验结果
- 批量重命名公共接口
- 大规模重组目录结构

### 6.3 Agent 严禁做的事

Agent 不得：

- 引入前视偏差
- 伪造或臆造缺失数据
- 为了让回测表现更好而偷偷改变口径
- 为了让测试通过而篡改测试意图
- 静默修改默认参数并不说明
- 删除无法复现的证据或历史输出
- 使用未来净值、未来标签、未来披露数据修补当前时点特征
- 在未说明假设的情况下编造基金字段含义

## 7. 数据规范

### 7.1 数据分层

建议将数据分为以下层级：

- `raw`：原始数据，尽量不可变
- `clean`：清洗与标准化后的数据
- `feature`：用于策略计算的特征数据
- `label`：回测或训练目标数据
- `result`：回测结果、分析结果、报告输出

### 7.2 数据基本要求

每个数据集应尽量明确以下信息：

- 数据来源
- 更新频率
- 主键字段
- 时间字段及其含义
- 披露滞后或可得性滞后
- 缺失值定义
- 单位与口径说明

### 7.3 时间字段规范

- 日期字段命名应统一，如 `trade_date`、`nav_date`、`report_date`、`available_date`
- 若存在“事件发生日”和“实际可获取日”，应分别保留
- 不得混淆“净值所属日期”和“数据抓取日期”

### 7.4 缺失值处理

Agent 不得默认以 0 替代缺失值，除非该字段定义允许这样做。
所有缺失值处理必须明确：

- 是删除样本
- 是前向填充
- 是分组内插值
- 是保留为空
- 是作为异常情况上报

### 7.5 原始数据不可手工修改

原始数据如需修正，应通过：

- 新增清洗规则
- 增加数据修复脚本
- 增加数据质量说明

而不是直接手工覆盖原始文件。

## 8. 基金领域特殊规则

### 8.1 基金池定义必须明确

任何策略实现都必须明确基金池构造规则，例如：

- 纳入哪些基金类型
- 是否剔除 ETF / LOF / QDII / FOF / 联接基金
- 是否要求基金成立满一定时间
- 是否设置最小规模门槛
- 是否要求净值历史足够长
- 是否剔除异常基金或数据严重缺失基金

### 8.2 A/C 份额处理

基金 A/C 份额、不同收费模式份额，默认不应被视为独立 alpha 来源。
除非明确说明，否则应遵循以下原则：

- 优先按基金实体或主基金聚合
- 避免同一基金不同份额重复入选
- 若保留不同份额，必须解释其业务原因和回测含义

### 8.3 清盘 / 合并基金处理

- 回测中应尽量保留历史上曾存在的基金
- 若基金清盘或合并，应明确退出逻辑
- 不得在历史样本中仅保留今天仍然存在的基金

### 8.4 新基金过滤

若策略要求基金成立满一定时间，必须：

- 明确时间窗口
- 在配置中体现
- 在回测和实盘模拟中保持一致

### 8.5 基金公司与经理暴露

若组合层面要控制基金公司、基金经理、风格或主题暴露，应：

- 明确规则
- 通过配置控制
- 在报告中展示约束影响

## 9. 因子与特征开发规范

### 9.1 因子实现原则

每个因子实现应尽量做到：

- 输入明确
- 输出明确
- 时间边界明确
- 与其他因子解耦
- 可单独测试

### 9.2 新增因子时必须说明

新增因子时，至少应说明：

- 因子名称
- 业务含义
- 输入数据来源
- 计算窗口
- 可获得性时点
- 是否需要去极值 / 标准化
- 是否需要行业/类型内比较
- 潜在风险：如过拟合、信息滞后、异常值敏感

### 9.3 因子预处理

若涉及以下操作，必须明确：

- winsorize / 去极值
- z-score 标准化
- 分层标准化
- 缺失值填充
- 中性化
- 排名转换

### 9.4 不允许的因子实现方式

- 直接使用未来收益构造当前特征
- 使用未来披露的持仓信息作为当前信号
- 在未说明情况下将结果导向信息混入输入特征

## 10. 策略变更检查清单

任何影响选基逻辑的改动，Agent 在输出总结时必须说明以下内容：

1. 基金池定义
2. 调仓频率
3. 信号生成时点
4. 执行时点
5. 因子定义
6. 因子预处理方式
7. 打分或排序方法
8. 组合构建方法
9. 风险约束
10. 成本假设
11. benchmark
12. 评价指标
13. 是否影响历史结果可比性

如果无法明确上述信息，Agent 应暂停关键策略改动并请求确认。

## 11. 回测规范

### 11.1 回测必须明确的要素

每次回测至少应明确：

- 回测起止日期
- 基金池定义
- 调仓频率
- 信号日期
- 执行日期
- 权重生成方式
- 成本假设
- benchmark
- 基金退出规则
- 缺失净值处理方式

### 11.2 交易与持仓口径

即使本项目不是实盘交易系统，也应尽量统一以下口径：

- 调仓日收盘后生成信号，下一可执行日生效，或其他明确约定
- 权重是否在调仓日等权、按分数权重、按风险平价等
- 是否考虑申购赎回费
- 是否考虑换手约束
- 是否考虑资金容量限制

### 11.3 异常情况处理

回测中若遇到以下情况，应明确处理：

- 基金净值缺失
- 基金暂停申购/赎回
- 基金清盘
- 新基金历史过短
- 数据源缺失导致当期无法评分

### 11.4 回测结果输出要求

每次回测输出应尽量记录：

- 策略名称
- 实验 ID
- git commit hash
- 配置文件路径
- 数据版本或快照标识
- 回测区间
- benchmark
- 主要绩效指标
- 输出文件路径

## 12. 实验管理规范

### 12.1 实验必须可追踪

每次实验应具备唯一标识，例如：

- `exp_20260318_factor_v2`
- `rank_momentum_6m_eqw_v1`

### 12.2 实验配置独立保存

实验应通过配置文件驱动，不应依赖手改源码中的参数。

### 12.3 历史结果默认不可覆盖

如无明确要求，不应覆盖已有实验结果。
默认行为应是创建新的输出目录。

### 12.4 必须保留对比基线

新实验应尽量与一个稳定 baseline 进行比较，并说明：

- 改动点
- 收益变化
- 风险变化
- 换手变化
- 是否更稳健

### 12.5 实验结论应区分“观察结果”和“因果结论”

Agent 可报告：

- “在当前样本期内该策略表现更好”

但不应武断报告：

- “该因子在未来一定有效”

## 13. 代码风格规范

### 13.1 通用原则

- 优先写清晰代码，而不是炫技代码
- 优先写可测试函数，而不是依赖全局状态
- 优先小函数组合，而不是超长流程函数
- 记得写清晰的注释

### 13.2 Python 规范

建议：

- 使用类型标注
- 公共函数提供 docstring
- 使用统一的 lint / format 工具
- 避免在业务逻辑中直接 `print`
- 使用结构化日志或统一日志封装

### 13.3 命名约定

命名应体现真实业务含义，避免：

- `tmp`
- `final_final`
- `new_logic2`
- `test_data2`

建议使用：

- `fund_universe`
- `rebalance_dates`
- `momentum_score`
- `portfolio_weights`

### 13.4 模块职责

- 因子模块只负责计算因子
- 回测模块只负责模拟组合历史表现
- 评估模块只负责分析与汇总结果
- 不要让一个模块同时负责“抓数据 + 算因子 + 回测 + 画图”

## 14. 测试规范

### 14.1 总体要求

新增业务逻辑必须尽量附带测试。
修改核心策略、回测或评分逻辑时，必须补充或更新测试。

### 14.2 测试类型

#### 单元测试

适用于：

- 因子计算函数
- 排序函数
- 基金过滤规则
- 指标计算
- 权重分配函数

#### 集成测试

适用于：

- 从输入数据到策略输出的最小闭环
- 从配置到回测结果的主流程

#### 回归测试

适用于：

- 核心策略在固定数据快照上结果不应无故漂移
- 关键指标在可接受范围内稳定

### 14.3 测试数据要求

- 尽量使用确定性样本
- 尽量使用小型 fixture
- 明确边界情况

### 14.4 必测边界情况

至少考虑以下情况：

- 空基金池
- 单只基金
- 净值缺失
- 历史长度不足
- 全部因子缺失
- 调仓日无可用基金
- 清盘基金在持仓期间退出

### 14.5 Bug 修复要求

若修复 bug，优先流程应为：

1. 编写能复现 bug 的测试
2. 修复代码
3. 验证测试通过
4. 说明根因与影响范围

## 15. 文档规范

以下变更应同步更新文档：

- 新增模块
- 新增因子
- 修改策略配置结构
- 修改回测口径
- 引入新数据源
- 修改实验输出格式

文档至少应说明：

- 目的
- 输入
- 输出
- 假设
- 限制
- 示例

## 16. Notebook 使用规范

`research/` 下的 notebook 仅用于研究探索，不应作为生产入口。

### 16.1 允许的用途

- 数据探索
- 初步因子验证
- 可视化分析
- 临时研究结论记录

### 16.2 不允许的用途

- 将 notebook 作为定时任务核心逻辑
- 把 notebook 里的单元作为唯一实现
- 在 notebook 中手工修改数据后直接产出正式结果

### 16.3 迁移要求

若 notebook 中逻辑需要进入主流程，应：

1. 提取为 `src/` 中的函数或类
2. 增加测试
3. 明确输入输出
4. notebook 仅保留示例调用

## 17. 默认工作流

Agent 在开始工作时，应尽量遵循以下流程：

1. 阅读 `README.md`、`AGENTS.md`、相关 `docs/`
2. 确认当前任务是“重构”“修 bug”“加因子”“改回测”还是“加数据源”
3. 先理解现有实现和依赖关系
4. 优先做最小必要改动
5. 修改后运行相关测试
6. 汇总变更内容、风险、假设、测试结果
7. 若涉及策略语义变化，明确说明对历史实验的影响

## 18. 常见任务操作手册

### 18.1 新增一个因子

建议步骤：

1. 在 `src/features/` 中新增因子实现
2. 明确输入字段、时间窗口、输出字段
3. 在配置中注册该因子
4. 增加单元测试
5. 增加最小样本集成测试
6. 更新因子文档
7. 用固定样本回测做基本验证
8. 汇报该因子是否改变策略默认行为

### 18.2 新增一个数据源

建议步骤：

1. 在 `src/data_ingestion/` 新增数据源模块
2. 明确字段定义、主键、更新时间与数据滞后
3. 增加数据质量检查
4. 更新文档与 schema 说明
5. 不直接污染现有原始数据
6. 若影响策略口径，必须说明

### 18.3 重构 notebook 逻辑到生产代码

建议步骤：

1. 确认 notebook 逻辑已稳定
2. 将逻辑拆解为纯函数或清晰模块
3. 加入类型标注与 docstring
4. 编写单元测试
5. notebook 改为调用 `src/` 模块
6. 验证结果与原 notebook 一致或说明差异

### 18.4 修改回测逻辑

建议步骤：

1. 明确改动是“修 bug”还是“改变口径”
2. 若改变口径，必须先说明影响范围
3. 增加回归测试
4. 记录对历史结果可比性的影响
5. 重新运行基线策略并输出差异摘要

### 18.5 新增组合约束

建议步骤：

1. 在 `portfolio/` 中新增约束逻辑
2. 通过配置控制阈值
3. 增加单元测试与边界测试
4. 在回测报告中增加约束影响分析
5. 说明对收益、回撤、换手的影响

## 20. 不确定时的处理原则

当需求不明确或现有代码语义不清晰时，Agent 应遵循以下原则：

1. 默认不改变策略语义
2. 默认不改变回测口径
3. 默认保留向后兼容
4. 默认先提出假设，再请求确认
5. 不要在金融假设上“自作聪明”
6. 可以留下清晰的 TODO / FIXME，但必须说明原因与风险

## 21. 优先级原则

当多个目标冲突时，优先级如下：

1. 金融逻辑正确性
2. 回测口径严谨性
3. 可复现性
4. 可测试性
5. 可维护性
6. 清晰性
7. 性能优化

## 23. 注释规范

注释的目标不是重复代码表面含义，而是帮助协作者理解：
- 业务背景
- 金融假设
- 时间边界
- 设计取舍
- 易错点
- 非显然实现原因
Agent 应将注释视为“长期协作接口”的一部分，而不是可有可无的附属品。

### 23.1 注释总原则

注释应满足以下原则：
1. 解释“为什么”，而不仅是“做什么”
2. 优先解释金融语义、时间边界和工程约束
3. 与代码保持同步
4. 简洁、准确、可验证
5. 不写过时、模糊、误导性注释

## 24. 最终要求

Agent 的所有改动应尽量满足以下标准：
- 代码可读
- 行为可测
- 输出可解释
- 结果可复现
- 风险可说明
- 偏差可控制
