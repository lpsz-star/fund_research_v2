# AGENTS.md

## 1. 目的

本文件用于规范 AI agents（如 Codex）与协作者在本仓库中的工作方式。

本项目是一个量化选基系统。我们希望所有改动都符合以下目标：

- 可复现
- 可测试
- 可审计
- 可持续迭代
- 避免量化研究中的常见偏差
- 保持研究逻辑与工程实现的一致性

Agent 在本仓库中的首要目标不是“尽快写出能跑的代码”，而是：

1. 保证金融逻辑正确
2. 保证回测口径严谨
3. 保证改动可验证、可追踪
4. 保证代码结构清晰、便于长期维护

## 2. 项目概述

本项目用于基金筛选、评分、组合构建、回测与评估，典型流程包括：

1. 获取并更新基金相关数据
2. 构建基金池（universe）
3. 计算基金因子 / 特征
4. 对基金进行打分与排序
5. 构建基金组合
6. 执行历史回测
7. 输出绩效评估与研究报告

## 3. 仓库结构约定

以下为推荐目录职责。若当前仓库结构与此不同，agent 应优先遵循现有结构，并在必要时提出结构优化建议，而不是擅自大幅重组。

```text
.
├─ src/
│  ├─ data_ingestion/     # 数据获取、更新、落库
│  ├─ data_processing/    # 数据清洗、标准化、对齐
│  ├─ universe/           # 基金池构建逻辑
│  ├─ features/           # 因子/特征计算
│  ├─ ranking/            # 打分、排序、筛选
│  ├─ portfolio/          # 组合构建与权重分配
│  ├─ backtest/           # 回测引擎
│  ├─ evaluation/         # 绩效指标与分析
│  ├─ reporting/          # 报表与可视化输出
│  └─ common/             # 通用工具、配置、日志等
├─ configs/               # 配置文件
├─ tests/                 # 单元/集成/回归测试
├─ research/              # 研究型 notebook，仅用于探索
├─ docs/                  # 设计文档、方法说明
├─ scripts/               # CLI 脚本、批处理入口
├─ data/                  # 数据目录（若保留在仓库外，应在 README 中说明）
├─ outputs/               # 回测结果、实验输出、报表
└─ AGENTS.md
```

## 4. 工程原则

以下原则优先于“代码简短”“实现速度快”。

### 4.1 正确性优先于性能

若正确性、可解释性、可复现性与性能冲突，优先保证前三者。

### 4.2 可复现优先

同样的代码版本、数据版本、配置文件，应尽可能得到一致结果。

### 4.3 严禁前视偏差

任何特征、标签、选基结果、回测收益，必须严格遵守时间边界。

### 4.4 尽量避免生存者偏差

基金池构建应尽可能考虑历史上已清盘、合并或不再存续的基金。

### 4.5 配置驱动

窗口长度、调仓周期、过滤阈值、评分权重、风险约束等参数应放在配置中，而不是散落在代码里硬编码。

### 4.6 分层清晰

数据、特征、评分、组合、回测、评估应尽量分层，避免单个函数承担过多职责。

### 4.7 小步迭代

Agent 应优先提交最小可验证改动，而不是一次性做大规模重构。

### 4.8 向后兼容

如果改动会影响历史实验的可比性，必须明确说明，并尽量保留旧逻辑入口或迁移说明。

## 5. Agent 行为边界

### 5.1 Agent 可以做的事

在无需额外确认的情况下，Agent 可以：

- 增加缺失的测试
- 补充文档
- 将硬编码参数配置化
- 修复明确的、可验证的 bug
- 改善日志、异常处理、类型标注

### 5.2 Agent 必须先确认再做的事

以下操作必须先征求用户确认：

- 修改默认策略逻辑
- 修改基金池定义
- 修改因子定义或打分口径
- 修改回测信号时点 / 执行时点
- 修改绩效指标计算方式
- 修改默认 benchmark
- 引入新的第三方数据源
- 删除历史实验结果
- 批量重命名公共接口
- 大规模重组目录结构

### 5.3 Agent 严禁做的事

Agent 不得：

- 引入前视偏差
- 伪造或臆造缺失数据
- 为了让回测表现更好而偷偷改变口径
- 为了让测试通过而篡改测试意图
- 静默修改默认参数并不说明
- 删除无法复现的证据或历史输出
- 使用未来净值、未来标签、未来披露数据修补当前时点特征
- 在未说明假设的情况下编造基金字段含义

## 6. 数据规范

### 6.1 数据分层

建议将数据分为以下层级：

- `raw`：原始数据，尽量不可变
- `clean`：清洗与标准化后的数据
- `feature`：用于策略计算的特征数据
- `result`：回测结果、分析结果、报告输出

### 6.2 数据基本要求

每个数据集应尽量明确以下信息：

- 数据来源
- 更新频率
- 主键字段
- 时间字段及其含义
- 披露滞后或可得性滞后
- 缺失值定义
- 单位与口径说明

### 6.3 时间字段规范

- 日期字段命名应统一，如 `trade_date`、`nav_date`、`report_date`、`available_date`
- 若存在“事件发生日”和“实际可获取日”，应分别保留
- 不得混淆“净值所属日期”和“数据抓取日期”

### 6.4 缺失值处理

Agent 不得默认以 0 替代缺失值，除非该字段定义允许这样做。
所有缺失值处理必须明确：

- 是删除样本
- 是前向填充
- 是分组内插值
- 是保留为空
- 是作为异常情况上报

### 6.5 原始数据不可手工修改

原始数据如需修正，应通过：

- 新增清洗规则
- 增加数据修复脚本
- 增加数据质量说明

而不是直接手工覆盖原始文件。


## 7. 因子与特征开发规范

### 7.1 因子实现原则

每个因子实现应尽量做到：

- 输入明确
- 输出明确
- 时间边界明确
- 与其他因子解耦
- 可单独测试

### 7.2 新增因子时必须说明

新增因子时，至少应说明：

- 因子名称
- 业务含义
- 输入数据来源
- 计算窗口
- 可获得性时点
- 是否需要去极值 / 标准化
- 是否需要行业/类型内比较
- 潜在风险：如过拟合、信息滞后、异常值敏感

### 7.3 因子预处理

若涉及以下操作，必须明确：

- winsorize / 去极值
- z-score 标准化
- 分层标准化
- 缺失值填充
- 中性化
- 排名转换

### 7.4 不允许的因子实现方式

- 直接使用未来收益构造当前特征
- 使用未来披露的持仓信息作为当前信号
- 在未说明情况下将结果导向信息混入输入特征

## 8. 策略变更检查清单

任何影响选基逻辑的改动，Agent 在输出总结时必须说明以下内容：

1. 基金池定义
2. 调仓频率
3. 信号生成时点
4. 执行时点
5. 因子定义
6. 因子预处理方式
7. 打分或排序方法
8. 组合构建方法
9. 风险约束
10. 成本假设
11. benchmark
12. 评价指标
13. 是否影响历史结果可比性

如果无法明确上述信息，Agent 应暂停关键策略改动并请求确认。

## 9. 回测规范

### 9.1 回测必须明确的要素

每次回测至少应明确：

- 回测起止日期
- 基金池定义
- 调仓频率
- 信号日期
- 执行日期
- 权重生成方式
- 成本假设
- benchmark
- 基金退出规则
- 缺失净值处理方式

### 9.2 交易与持仓口径

即使本项目不是实盘交易系统，也应尽量统一以下口径：

- 调仓日收盘后生成信号，下一可执行日生效，或其他明确约定
- 权重是否在调仓日等权、按分数权重、按风险平价等
- 是否考虑申购赎回费
- 是否考虑换手约束
- 是否考虑资金容量限制

### 9.3 异常情况处理

回测中若遇到以下情况，应明确处理：

- 基金净值缺失
- 基金暂停申购/赎回
- 基金清盘
- 新基金历史过短
- 数据源缺失导致当期无法评分

### 9.4 回测结果输出要求

每次回测输出应尽量记录：

- 策略名称
- 实验 ID
- git commit hash
- 配置文件路径
- 数据版本或快照标识
- 回测区间
- benchmark
- 主要绩效指标
- 输出文件路径

## 10. 实验管理规范

### 10.1 实验必须可追踪

每次实验应具备唯一标识，例如：

- `exp_20260318_factor_v2`
- `rank_momentum_6m_eqw_v1`

### 10.2 实验配置独立保存

实验应通过配置文件驱动，不应依赖手改源码中的参数。

### 10.3 历史结果默认不可覆盖

如无明确要求，不应覆盖已有实验结果。
默认行为应是创建新的输出目录。

### 10.4 必须保留对比基线

新实验应尽量与一个稳定 baseline 进行比较，并说明：

- 改动点
- 收益变化
- 风险变化
- 换手变化
- 是否更稳健

### 10.5 实验结论应区分“观察结果”和“因果结论”

Agent 可报告：

- “在当前样本期内该策略表现更好”

但不应武断报告：

- “该因子在未来一定有效”

## 11. 代码风格规范

### 11.1 通用原则

- 优先写清晰代码，而不是炫技代码
- 优先写可测试函数，而不是依赖全局状态
- 优先小函数组合，而不是超长流程函数
- 记得写清晰的注释

### 11.2 Python 规范

建议：

- 使用类型标注
- 公共函数提供 docstring
- 使用统一的 lint / format 工具
- 避免在业务逻辑中直接 `print`
- 使用结构化日志或统一日志封装

### 11.3 命名约定

命名应体现真实业务含义，避免：

- `tmp`
- `final_final`
- `new_logic2`
- `test_data2`

建议使用：

- `fund_universe`
- `rebalance_dates`
- `momentum_score`
- `portfolio_weights`

### 11.4 模块职责

- 因子模块只负责计算因子
- 回测模块只负责模拟组合历史表现
- 评估模块只负责分析与汇总结果
- 不要让一个模块同时负责“抓数据 + 算因子 + 回测 + 画图”

### 11.5 注释规范

注释的目标不是重复代码表面含义，而是帮助协作者理解：
- 业务背景
- 金融假设
- 时间边界
- 设计取舍
- 易错点
- 非显然实现原因
Agent 应将注释视为“长期协作接口”的一部分，而不是可有可无的附属品。

注释应满足以下原则：
1. 解释“为什么”，而不仅是“做什么”
2. 优先解释金融语义、时间边界和工程约束
3. 与代码保持同步
4. 简洁、准确、可验证
5. 不写过时、模糊、误导性注释


## 12. 测试规范

### 12.1 总体要求

新增业务逻辑必须尽量附带测试。
修改核心策略、回测或评分逻辑时，必须补充或更新测试。

### 12.2 测试类型

#### 单元测试

适用于：

- 因子计算函数
- 排序函数
- 基金过滤规则
- 指标计算
- 权重分配函数

#### 集成测试

适用于：

- 从输入数据到策略输出的最小闭环
- 从配置到回测结果的主流程

#### 回归测试

适用于：

- 核心策略在固定数据快照上结果不应无故漂移
- 关键指标在可接受范围内稳定

### 12.3 测试数据要求

- 尽量使用确定性样本
- 尽量使用小型 fixture
- 明确边界情况

### 12.4 必测边界情况

至少考虑以下情况：

- 空基金池
- 单只基金
- 净值缺失
- 历史长度不足
- 全部因子缺失
- 调仓日无可用基金
- 清盘基金在持仓期间退出

### 12.5 Bug 修复要求

若修复 bug，优先流程应为：

1. 编写能复现 bug 的测试
2. 修复代码
3. 验证测试通过
4. 说明根因与影响范围


## 13. 默认工作流

Agent 在开始工作时，应尽量遵循以下流程：

1. 阅读 `README.md`、`AGENTS.md`、相关 `docs/`
2. 确认当前任务是“重构”“修 bug”“加因子”“改回测”还是“加数据源”
3. 先理解现有实现和依赖关系
4. 优先做最小必要改动
5. 修改后运行相关测试
6. 汇总变更内容、风险、假设、测试结果
7. 若涉及策略语义变化，明确说明对历史实验的影响


## 14. 不确定时的处理原则

当需求不明确或现有代码语义不清晰时，Agent 应遵循以下原则：

1. 默认不改变策略语义
2. 默认不改变回测口径
3. 默认保留向后兼容
4. 默认先提出假设，再请求确认
5. 不要在金融假设上“自作聪明”
6. 可以留下清晰的 TODO / FIXME，但必须说明原因与风险


## 15. 文档要求

Agent 在完成修改时，应同步维护以下文档，便于 agent 与人类协作者协同：
- `README.md`：作为项目入口文档，当项目使用方式、结构、能力边界或关键模块发生变化时，应同步更新
- `docs/architecture.md`：系统架构
- `docs/data_dictionary.md`：数据字典
- `docs/backtest_conventions.md`：回测口径
- `docs/factor_catalog.md`：因子说明
- `docs/experiment_guide.md`：实验管理说明
- `docs/error_log.md`：记录历史错误、根因分析、修复方案、影响范围
- `docs/changes.md`：记录每次改动与变更摘要
